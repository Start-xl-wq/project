# 第一部分：物理灾难（为什么信号会变差？）
在 PCIe Gen3 的 8GT/s 速率下，主板上的铜线不再是普通的导线，而是“微波传输带”。高频信号在 PCB 板上跑的时候，会遇到严重的衰减和干扰。
## 1. 眼图 (Eye Diagram) —— 评判物理信号质量的“CT 片”
* 痛点：在 8GHz 的频率下，方波（0和1）经过主板走线后，尖角会被磨圆，变成正弦波甚至是一滩烂泥。前一个比特的电压还没降下来，后一个比特就来了，这叫 码间干扰 (ISI)。
* 什么是眼图：硬件工程师用高频示波器，把千百万个 0变1 和 1变0 的波形叠加在一起看。
* 如果波形非常清晰，中间会形成一个巨大的空白区域，形状像一只睁开的眼睛。眼睛睁得越大，说明 0 和 1 的界限越分明，接收端绝对不会认错。
* 如果信号太差，波形全糊在一起，中间的空白消失了，这就是“闭眼”。接收端就会把 1 认成 0。
* 总结：眼图是物理层的终极审判。眼图闭合，意味着后面的 LCRC 报错将会满天飞。
## 2. De-emphasis (去加重) —— 拯救眼图的“第一种魔法”
• 原理：假设你要连续发送 1, 1, 1, 1, 0。因为线路上有电容效应，连续发四个 1 会导致电压越积越高。当你想发 0 的时候，电压很难在极短的时间内“拉到底”，导致 0 被误认为是 1（眼图闭合）。
• 怎么做：芯片在发送第一个 1 时，用正常电压（比如 1.0V）。但发现后面几个也是 1，就主动降低电压（去加重，降到 0.7V）。这样积攒的电压就不会太高，等最后发 0 的时候，就能瞬间跌落到低电平。
• 总结：为了让信号跳变时更干脆，故意削弱连续相同信号的强度。这是 Gen1/Gen2 就在用的固定技术。
## 3. Equalization Preset (均衡预设 / EQ) —— Gen3 独有的“动态谈判”
* 痛点：到了 Gen3，信号太快了，固定的 De-emphasis 挡不住衰减。而且，有的 Wi-Fi 卡插在笔记本主板上（线很短），有的插在台式机延长线上（线很长），固定的补偿电压根本不适用。
* 什么是 Preset：PCIe 协议规定了 11 组预设值（Preset 0 ~ Preset 10）。每一组代表了一种不同强度的补偿方案（包含了不同的 De-emphas• is 去加重和 Preshoot 预冲）。
* 动态谈判 (LTSSM Recovery 阶段)：
    * 主板对 Wi-Fi 芯片说：“我先用 Preset 4 给你发一段测试码（TS1/TS2），你看看眼图睁得大不大？”
    * Wi-Fi 芯片回：“不行，高频衰减太厉害，你换 Preset 7 试试，加点劲！”
    * 经过几微秒的讨价还价，双方锁定一组误码率最低的 Preset，然后才正式进入 L0 状态开始通信。
* 工程踩坑：有些廉价的主板或 Wi-Fi 芯片，硬件 EQ 训练状态机有 Bug，死活谈不拢。结果就是 PCIe 降级到 Gen2 甚至 Gen1 运行，或者反复在 Recovery 状态横跳，连设备都枚举不到。
# 第二部分：链路层的挣扎（软件层面看到的现象）
当物理层的 EQ 没调好，或者主板走线太差导致“眼图微闭”时，0 和 1 就会发生反转。这时候，我们之前讲的 LCRC 和 Replay 就要登场救命了。
## 4. LCRC Error (Link CRC 错误) —— 沉默的守护者
* 发生什么了：Wi-Fi 芯片通过 DMA 向主板发送了一个 256B 的 MWr 报文。但是因为主板环境有电磁干扰，其中 1 个 bit 从 0 变成了 1。
* 硬件处理：
    a. 主板的 PCIe IP 核接收到报文，硬件计算出这 256B 的 CRC 校验码。
    b. 跟报文尾部自带的 4 字节 LCRC 一比对：不一致！
    c. 主板硬件立刻且静默地丢弃这个报文（绝不让错误数据进入系统内存），并向 Wi-Fi 芯片发送一个 NAK DLLP（拒绝应答）。
* 软件视角：驱动层面完全无感知，因为硬件拦下来了。但是在 Linux 的 dmesg 或 lspci -vvv 的 AER (Advanced Error Reporting) 寄存器中，你会看到 Bad TLP 或 Bad DLLP 的计数器在悄悄增加。这叫 Correctable Error (可纠正错误)。
## 5. Replay 增加的现象 —— 性能崩塌与掉卡的前兆
当 LCRC Error 发生，NAK 被发出后，Wi-Fi 芯片的 Replay Buffer（重传缓存） 就会疯狂运作。这就是“Replay 增加”。
作为驱动工程师，你会观察到以下三种典型的阶梯式现象：
* 现象 1：测速不达标，但不断网（轻度 Replay）
    * 本来 Wi-Fi 跑 802.11ac 应该有 800MB/s。但因为物理线路一直有小概率误码，底层 PCIe 在疯狂重传相同的报文。
    * 你的 PCIe 吞吐量被重传的“废包”占满了，留给真实网络包的带宽大减。测速只有 300MB/s。
* 现象 2：打游戏 Ping 值诡异跳动（中度 Replay）
    * PCIe 报文被 NAK 后重传，会产生几百纳秒到几微秒的延迟。
   * 如果连续发生重传，操作系统的网卡驱动 (NAPI) 拿不到数据，就会导致上层的 TCP/UDP 延迟出现毫无规律的毛刺（Jitter）。
* 现象 3：PCIe 链路崩溃，Wi-Fi 卡离奇消失（重度 Replay - 致命）
    * PCIe 协议规定：一个报文如果在 Replay Buffer 里重传了 4 次（通常是4次），仍然收到 NAK 或超时，硬件就会绝望。
    * 此时，链路层会触发一个 Replay Num Rollover 或者 Replay Timer Timeout 的致命错误 (Fatal Error)。
    * LTSSM 状态机会直接崩溃，强制跳回 Detect 状态重新握手。
    * 在操作系统的表现就是：设备突然抛出 Surprise Removal 中断，Linux 打印 PCIe link down，Wi-Fi 彻底断网，必须重启电脑才能恢复！
# 总结卡片：打通任督二脉
将这 5 个词连成一句话：

```c
因为主板走线质量差，高频信号衰减导致 【眼图 (Eye Diagram)】 闭合；
此时 PCIe Gen3 的 【Equalization Preset】 和 【De-emphasis】 机制试图补偿信号但失败了；
这导致接收端频繁收到错位的比特，触发了大量的 【LCRC Error】；
为了保证数据完整性，链路层的 【Replay 重传】 次数激增；
最终表现为 Wi-Fi 吞吐量暴跌，甚至重传超限导致 PCIe 链路断开（掉卡）。
```

**建议**：以后当调试 Wi-Fi，只要遇到“数据跑不满”、“高负载下掉卡”，第一时间不要去查 DMA Ring Buffer 代码，而是用命令（如 Linux 下的 lspci -vvv -s <BDF>）去看 PCIe AER (高级错误报告) 寄存器里的 Correctable Error Counters。如果看到 Replay 或 Bad TLP 计数器在狂飙，直接把锅甩给硬件工程师：“你们板子的 PCIe 信号质量有问题，去测眼图！”。
