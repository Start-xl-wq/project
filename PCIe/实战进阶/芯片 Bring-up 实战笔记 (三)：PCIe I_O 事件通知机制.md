> 在设备将状态写回主机内存（Write-Back）后，必须有一种机制通知 Host CPU。工业界根据不同的性能需求，演化出了 3
> 种截然不同的流派：

# 模式一：纯中断模式 (Pure Interrupt)
【核心逻辑】 兼职打工模式 —— “有事叫我，没事我干别的”。
* 工作流：
    a. Device 将状态写回主机内存。
    b. Device 发送 PCIe MWr TLP 包，触发 MSI / MSI-X 中断。
    c. CPU 被打断当前正在执行的进程。
    d. 操作系统保存上下文 -> 切换到内核态 -> 执行硬件中断处理 -> 触发软中断 -> 回收数据 -> 恢复上下文。
* 致命缺点：在极高吞吐量（如万兆以上网卡）下，每秒产生数百万次中断。极高的上下文切换开销（每次 1~3 微秒）会导致 中断风暴（Interrupt Storm），系统会因为无休止地处理中断而彻底卡死（Livelock）。
* Linux 现状与业务场景：
    * 现状：已退出高速网络领域，但依然是 Linux 的基础机制。
    * 场景：低带宽、低并发、极度看重 CPU 多任务复用和省电的场景。
    * 代表：个人 PC / 笔记本电脑、树莓派等 IoT 边缘节点、传统 SATA 硬盘、鼠标键盘等慢速外设。
---
# 模式二：纯轮询 / 内核旁路模式 (Pure Polling / Kernel Bypass)
【核心逻辑】 霸道总裁模式 —— “买断你的全部时间，死等我的消息”。
• 工作流：
    a. 完全关闭 PCIe 设备的 MSI 中断。
    b. 操作系统将网卡控制权直接交给用户态程序（如 DPDK / SPDK），这被称为 内核旁路（Kernel Bypass）。
    c. 剥离出一个或多个独立的物理 CPU 核心（绑核），跑一个纯正的 while(1) 死循环。
    d. 借助 Intel DDIO 等技术，CPU 疯狂读取本地 Cache，一旦 Device 写回状态，CPU 在几纳秒（ns）内瞬间感知并处理。
* 致命缺点：极度浪费 CPU 算力。无论有没有流量，这个被绑定的 CPU 核心使用率永远是 100%，处于疯狂的空转状态。
* Linux 现状与业务场景：
    * 现状：脱离了 Linux 内核的管辖，属于高阶玩家的自定义玩法。
    * 场景：不差钱、不惜浪费成百上千个 CPU 核心空转，只为换取极限吞吐量和极低延迟的顶级性能场景。
    * 代表：
       * 金融高频交易 (HFT)：微秒必争的量化交易系统。
       * 5G 核心网 (NFV) / 云计算底层虚拟路由器：处理海量零碎小包的基础设施。
       * 高端 AI / HPC 算力集群底层存储：榨干 NVMe 全闪存阵列 IOPS 极限的 SPDK 系统。
---
# 模式三：NAPI 混合模式 (Interrupt + Polling 动态切换)
【核心逻辑】 弹性伸缩模式 —— “平时休眠省电，流量爆发时化身无情收包机器”。
* 工作流（Linux 巧妙的折中智慧）：
    a. 平时闲着：网络接收线程休眠，等待 MSI 中断。
    b. 首包到达：Device 触发一次中断，唤醒 CPU。
    c. 屏蔽中断（关键）：CPU 醒来后，立刻在软件层面 关闭网卡中断，主动切换为轮询模式（Polling）。
    d. 大流量轮询：在一次轮询周期内（Budget），疯狂且高效地批量收取 Descriptor Ring 里的数据。
    e. 恢复原状：当 Descriptor Ring 被抽空（说明这波流量洪峰过去了），CPU 重新开启网卡中断，并恢复休眠。
* 优势：既避免了“中断风暴”导致系统瘫痪，又避免了“纯轮询”导致 CPU 100% 烧毁空转。
* Linux 现状与业务场景：
    * 现状：目前 Linux 网络子系统的绝对主流标准。 千兆/万兆/25G 网卡的原生 Linux 驱动默认全部采用此框架。
    * 场景：兼顾性能与 CPU 资源的绝大多数企业级服务端应用。
    * 代表：淘宝/微信的普通业务服务器、MySQL/Redis 数据库节点、Nginx 反向代理、公有云（AWS/阿里云）卖给普通用户的标准云主机。
---
# 💡 一句话总结
* 模式 1（纯中断）：省钱省电，给个人电脑和慢速外设用（重分时复用）。
* 模式 3（NAPI 混合）：极佳的工程平衡，Linux 服务器的默认标配，给常规互联网大厂的后台业务用。
* 模式 2（纯轮询）：暴力美学，绕开 Linux 内核，专给金融极速交易、5G 基站和 AI 底层算力中心等“不差钱只求快”的土豪场景用。
